# Harness-First Agentic Development Method

## 1. 文档目的

这份文档定义 OfferPilot 当前采用的正式开发方法。

目标不是提出一个抽象概念，而是沉淀一套可以长期复用、可迁移到其他项目、并且适合当前协作关系的方法论。

这套方法同时满足以下要求：

1. 用户本人不需要懂代码，也不需要承担架构设计责任
2. 开发过程需要由 agent 端到端承担工程执行
3. 需求通常不是一开始就明确，需要先收敛再实现
4. 每次交付前都必须经过测试、构建、验证，而不是直接交给用户碰运气
5. 用户应当像真实用户一样试用产品，再给体验反馈
6. 方法需要能复用到未来其他 AI 产品或普通软件项目中

## 2. 方法定义

OfferPilot 采用一种混合式开发方法：

**Harness-First Agentic Development**

中文可表述为：

**以 Harness 为主、以 Agentic 为范式、以 Ralph-style loop 为局部闭环机制的混合开发法。**

这个定义不是把三个词并排堆砌，而是明确它们在系统中的层级关系。

### 2.1 上层：Agentic Engineering

`Agentic Engineering` 负责定义协作关系。

它回答的是：

- 谁负责什么
- 人和 agent 如何分工
- 工程执行如何在保持人类控制权的前提下交给 agent

在本项目里，Agentic 层的基本原则是：

1. 人类负责目标、优先级、体验判断和最终验收
2. Agent 负责方案收敛、实现、测试、部署和修复
3. 人类不承担底层技术实现义务
4. Agent 不擅自改变产品方向，必须在既定目标内执行

因此，Agentic 在这里是**协作范式**，不是日常执行细则。

### 2.2 中层：Harness Engineering

`Harness Engineering` 是默认执行框架。

它回答的是：

- 如何把 agent 的产出纳入稳定流程
- 如何减少不可控输出
- 如何通过环境、约束、验证和反馈来提高交付质量

在本项目里，Harness 是主方法，因为真正决定开发质量的不是“用了什么模型”，而是有没有把以下环节设计清楚：

- 需求入口
- 文档约束
- 数据和接口边界
- schema 和 prompt 约束
- 测试与构建
- 部署与回归验证
- 用户反馈回流

因此，OfferPilot 的默认说法应当是：

> 我们采用 Harness-First 的 Agentic Development。

### 2.3 下层：Ralph-style Loop

`Ralph Wiggum Loop` 不作为全局开发方法，而作为局部递归修复机制存在。

它适用的场景包括：

1. 明确 bug 修复
2. 小范围重构
3. schema / prompt 兼容问题
4. 文档和代码不一致的修补

它不适合直接用来主导：

1. 产品定位调整
2. 页面信息架构设计
3. 跨模块架构重构
4. 需要业务取舍和用户理解的重大决策

因此，在本方法论里，Ralph-style loop 是一个**局部执行子机制**，只在问题清晰、边界明确时使用。

## 3. 为什么必须是混合而不是单选

如果只采用 `Agentic Engineering`，会有两个问题：

1. 定义了分工，但没有定义质量控制
2. 容易变成“人提一句，agent 自由发挥”

如果只采用 `Harness Engineering`，也会有问题：

1. 它更像执行框架，而不是对协作角色的完整描述
2. 不足以表达“用户不懂代码也可以驱动项目”的模式

如果只采用 `Ralph Loop`，问题更明显：

1. 它过于偏向递归修复
2. 不适合用来定义整个项目从需求到上线的流程

所以三者的正确组合方式是：

- `Agentic`：定义人和 agent 的协作范式
- `Harness`：定义开发、测试、部署、反馈的主流程
- `Ralph-style loop`：定义小问题如何快速闭环

## 4. 项目内正式角色定义

为了让这套方法可复用，角色定义必须明确而稳定。

### 4.1 人类产品 Owner

在 OfferPilot 中，人类产品 Owner 的职责是：

1. 提出目标、痛点和想要的体验
2. 判断优先级
3. 对功能是否“真的有用”做主观验收
4. 以真实用户身份试用产品
5. 反馈体验、问题和预期差

明确不要求 Product Owner 负责：

1. 写代码
2. 设计数据库
3. 设计系统架构
4. 写测试
5. 排查部署问题
6. 决定具体工程实现方式

这保证了不懂代码的人也可以有效驱动产品。

### 4.2 Agent 工程负责人

在 OfferPilot 中，agent 的职责是：

1. 把模糊需求收敛成可执行方案
2. 维护文档一致性
3. 完成前后端实现
4. 完成 AI workflow、schema、prompt、数据层设计
5. 自己先做测试和验证
6. 完成部署、修复和回归验证
7. 把用户反馈转化为下一轮产品迭代

明确限制：

1. Agent 不能擅自偏离已确认目标
2. Agent 不能把未经验证的实现直接交付为“已完成”
3. Agent 必须在文档、代码、测试和部署之间保持一致性

### 4.3 用户验收者

在 OfferPilot 中，Product Owner 同时也是真实种子用户。

因此验收方式不是代码 review，而是：

1. 像真实用户一样登录和使用
2. 关注流程是否顺畅
3. 关注文案是否清楚
4. 关注结果是否可信
5. 关注等待时间和交互负担

这条规则对于未来其他项目也成立：  
**最终体验验收应该尽量站在用户视角，而不是工程视角。**

## 5. 默认开发闭环

这套方法的默认工作流分为 6 个阶段。

### 阶段 1：需求收敛

输入通常是模糊的，例如：

- 我希望这个流程更简单
- 我想让它更像真实用户会用的产品
- 我觉得这个地方太麻烦

此时 agent 的任务不是立即写代码，而是先把需求收敛成：

1. 目标用户是谁
2. 用户真正要完成什么任务
3. 这次迭代解决什么，不解决什么
4. 成功标准是什么
5. 有没有现有文档需要同步

阶段输出应当是：

- 口头共识或文档补丁
- 主路径定义
- 范围边界

### 阶段 2：方案翻译

一旦需求清楚，agent 负责把它翻译成可执行方案。

标准输出包括：

1. 页面或交互变化
2. 数据结构变化
3. AI workflow 变化
4. API 变化
5. 验收标准
6. 风险和已知约束

如果改动较大，应形成新文档或计划文档。

### 阶段 3：实现

agent 负责端到端推进实现，通常覆盖：

1. 前端页面和交互
2. 后端 API
3. 数据库 schema / migration
4. service layer
5. prompt / schema / model routing
6. 部署配置

原则：

1. 先打通最短主路径
2. 先做稳定结构化输出，再做 fancy 交互
3. 不因为炫技而扩展范围

### 阶段 4：工程验证

在交给用户前，agent 必须先完成一轮工程验收。

最低要求：

1. 补或更新相关测试
2. 跑 targeted tests
3. 跑 full test suite
4. 跑 build
5. 做必要的 smoke check

只有在这些通过后，才可以宣称一项功能“已完成，可试用”。

### 阶段 5：用户试用验收

用户试用时，不要求看代码。

用户只需要关注：

1. 流程是否自然
2. 页面是否直观
3. 输出是否值得信任
4. 响应速度是否可接受
5. 功能是否真的帮助完成目标任务

这一步的输出是体验反馈，而不是技术方案。

### 阶段 6：反馈回流

用户反馈后，agent 再把反馈转成下一轮迭代任务。

反馈通常分为三类：

1. 体验类：太麻烦、太慢、不够直观
2. 质量类：结果不准、不可信、文案有问题
3. 范围类：需要新增或收缩能力

回流后，重新进入阶段 1。

## 6. Harness 的具体组成

为了让方法能落地，Harness 必须具象化。

### 6.1 产品 Harness

作用：约束方向，不让项目跑偏。

包含：

- Product Context
- PRD
- Journey
- phase plan
- 当前主路径定义
- 非目标列表

规则：

1. 新需求优先检查是否破坏当前主路径
2. 范围变化必须同步关键文档
3. 不允许口头共识长期悬空不落文档

### 6.2 工程 Harness

作用：约束实现边界。

包含：

- AGENTS.md
- 目录结构
- 命名规范
- 页面语言规则
- 数据边界
- 提交和部署规则

规则：

1. 所有工程产物必须遵守仓库约束
2. 项目结构必须保持清晰可维护
3. 不能为了临时跑通而长期堆积隐性例外

### 6.3 AI Harness

作用：约束模型行为，降低幻觉和格式失控。

包含：

- prompt 模板
- zod schema
- model routing
- fallback 策略
- structured outputs
- latency / token 预算

规则：

1. 不允许把关键输出完全交给自由文本
2. 结构化结果必须可校验
3. 对不同任务使用不同模型，而不是一把梭

### 6.4 质量 Harness

作用：保证交付不是“看起来能跑”。

包含：

- 单元测试
- route tests
- integration tests
- build
- smoke tests
- regression tests

规则：

1. 修 bug 时优先补回归测试
2. 声称完成前必须有验证证据
3. 测试不过不能当成功能完成

### 6.5 部署 Harness

作用：保证本地和线上不脱节。

包含：

- GitHub 仓库
- Vercel 自动部署
- Supabase migration
- env 管理
- 线上 smoke 验证

规则：

1. 每次有实质进展必须提交并推送
2. 环境变量和 migration 必须与线上一致
3. 线上失败不等于功能完成

### 6.6 反馈 Harness

作用：让真实使用体验进入下一轮迭代。

包含：

- 用户报错
- 用户试用感受
- 主观体验反馈
- 线上异常信息
- 回归场景沉淀

规则：

1. 体验问题不能被简单归类为“UI 不满意”
2. 反馈要转成可执行问题
3. 重大反馈要更新文档和主路径认知

## 7. Ralph-style Bugfix Loop

这是本方法中唯一允许高频递归执行的局部 loop。

适用场景：

1. 明确报错
2. 可复现失败
3. 边界清晰的兼容问题
4. 小范围逻辑错误

标准步骤：

1. 复现问题
2. 定位根因
3. 写 failing test 或补回归测试
4. 做最小修复
5. 跑 targeted tests
6. 跑 full tests / build
7. 必要时做线上 smoke check
8. 提交、推送、记录

不允许的误用方式：

1. 把产品方向讨论也做成递归 loop
2. 没有复现就盲修
3. 没有测试就频繁试错
4. 把多个不相关问题混成一个 loop

## 8. 决策优先级规则

为了避免方法前后矛盾，必须定义清楚优先级。

发生冲突时，按以下顺序决策：

1. **用户主任务优先**
   如果技术上很酷，但会破坏主任务完成率，不做。

2. **主路径优先于辅助能力**
   优先保证用户能完成核心闭环，再做侧边能力。

3. **稳定性优先于花哨**
   先保证正确和可控，再谈复杂体验。

4. **验证过的实现优先于猜测**
   凡是可以通过测试、构建、线上验证确认的，都不要靠“应该没问题”。

5. **反馈优先于自我想象**
   用户试用反馈高于工程师主观偏好。

6. **文档一致性优先于局部便利**
   重要目标变化时，必须同步 Context / Journey / Plan。

## 9. 这套方法在 OfferPilot 中的具体应用

OfferPilot 当前的真实执行方式如下：

### 用户负责

1. 提出目标和产品想法
2. 判断优先级
3. 进行真实试用
4. 反馈体验

### Agent 负责

1. 收敛需求
2. 更新文档
3. 设计技术方案
4. 实现前后端和 AI workflow
5. 跑测试和构建
6. 处理部署与 bug

### 验收方式

1. agent 先做工程验证
2. 用户再做使用验收
3. 验收反馈进入下一轮

这意味着 OfferPilot 的开发并不是：

- 用户写详细技术规格，agent 照做

而是：

- 用户给目标和反馈
- agent 负责完成从模糊需求到可用产品的完整闭环

## 10. 复用到其他项目时怎么改

这套方法可以直接复用到未来项目，但需要替换 4 个变量。

### 10.1 替换产品主路径

OfferPilot 的主路径是：

`登录 -> 上传简历 -> 提交 JD -> 改写建议 -> 模拟面试`

未来项目只需要替换成新的主路径。

### 10.2 替换用户主任务

要重新定义：

1. 用户最核心要完成的任务是什么
2. 成功标准是什么
3. 哪些能力属于主路径，哪些属于支撑层

### 10.3 替换 AI Harness

不同项目的 prompt、schema、model routing 会变。

但这套原则不变：

1. 有结构化输出
2. 有模型职责分工
3. 有 guardrails
4. 有验证方式

### 10.4 替换反馈来源

未来项目的反馈可能来自：

- 用户试用
- 运营数据
- B 端客户
- 销售反馈
- 内部使用者

但都必须进入同一套回流机制。

## 11. 新项目启动模板

如果未来复用这套方法，新项目应按这个顺序开始。

1. 写 Product Context
2. 写主路径定义
3. 明确用户角色和 agent 角色
4. 明确 non-goals
5. 明确 harness 组成
6. 再进入实现

新项目的最小文档模板建议至少包含：

1. `Product Context`
2. `Project Journey`
3. `PRD`
4. `Development Method`
5. `Session Handoff Prompt`

## 12. 反模式清单

以下做法被视为违背本方法。

1. 用户提出体验问题后，直接跳到技术实现而不先收敛需求
2. agent 未经验证就宣称完成
3. 文档和当前代码状态长期不一致
4. 用“Agent 很强”替代明确的边界和测试
5. 遇到 bug 不先复现和写测试，而是盲目试错
6. 让用户承担不必要的技术决策
7. 把底层能力暴露给用户，导致主路径变复杂

## 13. 一句话对外描述

如果未来需要对外介绍这套方法，推荐使用以下表述：

**我们采用一种 Harness-First 的 Agentic Development 方法：由人类负责需求、优先级和体验验收，由 agent 负责端到端工程执行，并通过局部 Ralph-style bugfix loop 保证问题快速闭环。**

## 14. 当前执行承诺

从本文件建立起，OfferPilot 的默认协作方式固定为：

1. 先收敛需求，再实现
2. 由 agent 负责端到端开发与验证
3. 由用户负责真实试用与体验反馈
4. 通过文档、测试、部署和反馈形成稳定闭环

除非明确修改本方法论文档，否则后续项目推进默认遵循本规范。
