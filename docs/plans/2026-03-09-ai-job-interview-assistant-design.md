# AI 求职面试助手项目设计

## 1. 项目目标

这个项目不是为了做一个“花哨的 AI demo”，而是为了同时实现 4 个目标：

1. 做出一个能写进简历的 AI 产品项目
2. 在做项目的过程中，把 AI PM 高频面试知识点真正用一遍
3. 在面试时能同时讲产品思维、AI 理解、评测、指标和业务判断
4. 做成你自己能长期使用、持续迭代的产品，而不是一次性作业

所以这个项目的设计原则是：

- `小而完整`
- `业务闭环清晰`
- `AI 价值明确`
- `便于讲故事`
- `2-3 周内能做出 MVP`

## 2. 方向选择：为什么选“AI 求职面试助手”

### 备选方向

#### 方案 A：AI 简历优化器

优点：

- 最容易做
- 用户需求明确
- 能快速出结果

缺点：

- 太像单点工具
- 不够覆盖 Agent / Workflow / 评测 / 状态管理
- 面试里容易被认为“只是一个 prompt 包装器”

#### 方案 B：AI 模拟面试官

优点：

- 很贴求职场景
- 容易讲多轮交互

缺点：

- 如果只有问答，还是偏窄
- 不够体现完整产品能力

#### 方案 C：AI 求职面试助手

包含：

- JD 解析
- 简历匹配
- 面试题生成
- 模拟面试
- 答案评估
- 知识库问答
- 进度管理

优点：

- 最容易形成完整产品闭环
- 能自然覆盖 RAG、Workflow、Agent、Memory、评测、指标、MVP、护栏指标
- 非常适合写进简历和面试中讲

缺点：

- 需要更强的范围控制

### 推荐结论

推荐做 `方案 C`，但必须严格收范围。  
不要做成“万能求职平台”，而要做成：

**面向 AI/产品岗求职者的 AI 面试准备 Copilot**

一句话定义：

`帮助用户把 JD、简历、面经、知识点和模拟面试流程串起来的 AI 求职面试助手。`

## 3. 产品定位

### 产品名称

可以暂定为：

- `OfferPilot`
- `面试 Copilot`
- `AI 求职面试助手`

### 目标用户

第一阶段只服务一类用户：

**正在准备 AI 产品经理 / 产品经理 / AI 应用类岗位面试的求职者**

你自己的角色就是第一批核心用户。

### 核心价值

这个产品的价值不是“帮用户生成更多字”，而是：

1. 把零散的面试资料整理成结构化备战路径
2. 根据 JD 和用户经历生成个性化准备内容
3. 把“知识点学习”变成“真实模拟”
4. 让用户在反复练习中看到进步和薄弱项

## 4. 用户痛点

这个项目最好讲的地方，是用户痛点非常真实。

### 典型痛点

1. 面试资料太散
2. 知识点看了很多，但不会答
3. 不知道该优先准备什么
4. 简历和岗位不匹配，缺少针对性包装
5. 模拟面试成本高，无法持续获得反馈
6. 练过的问题没有沉淀，下一次还得从头来

### 你要解决的核心问题

**如何让用户围绕一个目标岗位，把“资料整理 -> 知识理解 -> 项目包装 -> 模拟问答 -> 复盘沉淀”串成闭环。**

## 5. 产品方案总览

## MVP 核心主线

MVP 只做 4 个核心功能：

1. `JD 解析与能力拆解`
2. `个性化题库与知识库问答`
3. `模拟面试与答案评估`
4. `面试记忆与薄弱项追踪`

这 4 个功能已经足够覆盖你要学的绝大多数 AI PM 高频知识点。

### 功能 1：JD 解析与能力拆解

输入：

- 岗位 JD

输出：

- 岗位要求拆解
- 高频考察维度
- 推荐准备重点

价值：

- 把抽象 JD 翻译成准备清单

### 功能 2：个性化题库与知识库问答

输入：

- 目标岗位
- 面经资料
- 知识库文档
- 用户简历/项目

输出：

- 个性化高频问题
- 某道题的解释
- 某个概念的结构化说明

价值：

- 让用户可以基于自己的目标岗位和知识库学习

### 功能 3：模拟面试与答案评估

输入：

- 用户选择岗位 / 问题类型
- 用户回答文本

输出：

- 结构化评分
- 不足点
- 改写建议

价值：

- 从“知道知识点”变成“会回答”

### 功能 4：面试记忆与薄弱项追踪

记录：

- 用户练过哪些题
- 哪些题答得差
- 哪类能力还弱
- 哪些表达反复出问题

价值：

- 形成持续训练闭环

## 6. 为什么这个项目天然覆盖 AI PM 高频考点

这个项目最大的优点，是它不是“为了学概念而学概念”，而是做着做着就会把高频八股都用掉。

### 1. Agent

这个项目天然能讲 Agent：

- 模拟面试是多轮任务
- 需要围绕目标持续推进
- 需要记住用户背景、目标岗位和前文表现

你可以讲：

`我没有把整个产品做成纯问答机器人，而是将部分环节 Workflow 化，将多轮模拟和个性化反馈设计成轻 Agent 式流程。`

### 2. Workflow

项目里最适合 Workflow 的环节：

- JD 解析
- 题库生成
- 回答评分
- 复盘总结

你可以讲：

`高频标准化任务优先使用 Workflow，而不是一开始就全自动 Agent。`

### 3. RAG

项目里可以接入：

- 面经文档
- 岗位 JD
- 你的知识库
- 你自己的项目笔记

你可以讲：

`我用 RAG 解决“岗位要求、面经和知识点分散且不断更新”的问题。`

### 4. Prompt

这个项目会迫使你设计：

- JD 拆解 prompt
- 面试官提问 prompt
- 回答打分 prompt
- 改写建议 prompt

你可以讲：

`Prompt 在这个项目里不是单独的技巧，而是贯穿任务定义、输出格式和评分标准的一层。`

### 5. Memory

这正是这个项目区别于普通问答 bot 的关键。

你可以设计两类 memory：

#### 用户长期记忆

- 目标岗位
- 城市偏好
- 项目经历
- 简历版本
- 常见薄弱项

#### 任务状态记忆

- 当前正在练什么题
- 上一次回答得分
- 哪些知识点反复出错
- 当前面试轮次和上下文

这让你能自然讲清楚：

- 为什么 Agent 需要外部记忆
- Memory 和 RAG 的区别

### 6. 评测

这个项目非常适合讲评测。

你可以把评测分成三层：

#### 知识问答层

- 回答是否覆盖关键点
- 是否结构清晰

#### 模拟面试层

- 回答是否切题
- 是否有逻辑
- 是否有业务感

#### 产品层

- 用户是否完成一轮完整训练
- 练习后回答质量是否提升

### 7. 指标

这个项目能自然覆盖：

- 北极星指标
- 护栏指标
- 留存
- 漏斗
- MVP

详见后面指标设计。

## 7. MVP 功能列表

## 页面/模块建议

### 模块 1：岗位录入页

### 模块 2：我的背景页

- 简历

### 模块 4：模拟面试页

支持：

## 8. 系统架构

## 设计原则

- 尽量本地优先
- 简单可实现
- 能讲清楚技术方案，不追求花哨

## 推荐架构

### 前端

- Web 前端即可
- 模块化页面：岗位、知识库、模拟面试、复盘

### 后端逻辑层

分为 4 层：

1. `JD 解析层`
2. `知识检索层`
3. `模拟面试层`
4. `用户记忆与状态层`

### AI 层

建议组合：

- `RAG`：岗位 JD、面经、知识点、项目笔记
- `Workflow`：JD 拆解、评分、复盘
- `轻 Agent`：多轮模拟面试

### 存储层

- 用户 profile
- 练习记录
- 错题/薄弱项
- 面试知识库索引

## 9. 数据流

### 场景一：用户输入 JD

1. 用户粘贴 JD
2. 系统解析岗位要求
3. 系统输出能力维度
4. 系统推荐优先准备内容

### 场景二：用户提问知识点

1. 用户提出问题
2. 系统检索知识库
3. 组装上下文
4. 输出结构化回答

### 场景三：模拟面试

1. 用户选择岗位和题型
2. 系统从题库和记忆中生成问题
3. 用户作答
4. 系统打分并给改写建议
5. 结果写回用户状态

## 10. 评测设计

这是这个项目最重要的加分点之一。

### 评测目标

不是评估“模型是不是聪明”，而是评估：

- 产品是否帮用户更好准备面试

### 第一层：知识问答效果

指标：

- 回答覆盖率
- 结构清晰度
- 引用相关性

### 第二层：模拟面试效果

指标：

- 回答评分稳定性
- 建议可执行性
- 用户是否能据此优化答案

### 第三层：产品价值效果

指标：

- 完成一轮训练的用户比例
- 连续练习率
- 薄弱项减少情况

## 11. 北极星指标与护栏指标

### 北极星指标

推荐定义：

**每周完成有效面试训练任务的用户数**

为什么不用“聊天次数”：

- 聊天次数不代表用户获得了价值

为什么这个指标更好：

- 它更接近“真的完成了一次有效准备”

### 护栏指标

- 错误回答率
- 幻觉率
- 用户放弃率
- 平均响应时长
- 反馈差评率

## 12. MVP 与后续迭代边界

### MVP 不做什么

不要做：

- 自动投递
- 全流程岗位爬虫
- 视频面试分析
- 复杂语音交互
- 社区和社交系统

### MVP 只验证什么

验证三个问题：

1. 用户是否真的会持续使用
2. 个性化题库和模拟是否有价值
3. RAG + 状态记忆 + 结构化评估是否能形成闭环

### 第二阶段再考虑

- 语音模拟面试
- 更强的多轮 Agent
- 多岗位迁移
- 学习路径推荐

## 13. 风险与兜底

### 风险 1：AI 评分不可靠

解决：

- 结构化评分 rubric
- 多维评分
- 提示“仅供参考”

### 风险 2：知识库检索不准

解决：

- 规范文档切分
- 提示引用来源
- 无法回答时明确说明

### 风险 3：产品范围太大

解决：

- 只做 AI 产品/产品岗求职
- 只做文字输入
- 只做核心 4 模块

## 14. 简历写法

项目名建议：

**AI 求职面试助手（RAG + Workflow + Memory）**

简历表达示例：

- 设计并实现面向 AI/产品岗求职者的 AI 面试准备助手，围绕 `JD 解析、知识库问答、模拟面试、薄弱项追踪` 搭建完整 MVP。
- 基于岗位 JD、面经文档与知识库内容构建 RAG 检索链路，提升岗位要求解析和问题解答的针对性与可信度。
- 将标准化任务设计为 Workflow，将多轮模拟与个性化反馈设计为轻 Agent 流程，并引入用户长期记忆与任务状态记录。
- 设计产品核心指标与评测框架，从回答覆盖率、模拟训练完成率、建议可执行性等维度验证产品价值。

## 15. 面试时怎么讲这个项目

你后面讲这个项目时，建议按下面结构：

1. 先讲背景：为什么要做
2. 再讲用户痛点
3. 再讲为什么这是 AI 适合切入的场景
4. 再讲产品方案
5. 再讲 AI 方案拆解：
   - 哪部分用 RAG
   - 哪部分用 Workflow
   - 哪部分用 Memory
6. 再讲指标和评测
7. 最后讲复盘和下一步

### 一个可直接复述的版本

`我做了一个面向 AI/产品岗求职者的 AI 面试准备助手，核心想解决的是求职资料分散、知识点看了不会答、练习没有沉淀这三个问题。产品上我先收敛到 JD 解析、知识库问答、模拟面试和薄弱项追踪四个模块。技术方案上，我没有把它做成一个单纯聊天机器人，而是把知识问答部分设计成 RAG，把 JD 拆解和复盘设计成 Workflow，把多轮模拟和个性化反馈做成轻 Agent 式流程，并加入用户长期记忆和任务状态记录。这样这个项目既能体现 AI 能力，也能体现产品思维、评测、指标和系统设计。`

## 16. 你做这个项目时会自然学会什么

这个项目最值钱的地方，是你做完之后，很多八股不用单独背了。

你会自然学会：

- 什么是 Agent
- 什么是 Workflow
- 为什么需要 RAG
- Memory 和 RAG 的区别
- 怎么做产品指标
- 怎么做 MVP
- 怎么做评测
- 怎么讲一个完整的 AI 产品故事
