# AI 求职面试助手 PRD v1

## 1. 文档目的

本文档用于定义 `AI 求职面试助手` 的第一版产品方案，明确：

- 产品目标
- 核心用户
- 核心场景
- MVP 功能范围
- 信息架构
- AI 方案拆解
- 指标与评测
- 非目标范围

这版 PRD 的目标不是追求“大而全”，而是指导一个可在短时间内完成、能写进简历、能讲进面试的 MVP。

## 2. 产品概述

### 2.1 产品名称

暂定名：

- `AI 求职面试助手`
- 英文工作名：`OfferPilot`

### 2.2 一句话定义

**一个面向中文求职用户的 AI 求职准备工作台，帮助用户完成 `登录 -> 上传并保存简历 -> 保存目标 JD -> 生成改写建议 -> 模拟面试`。**

### 2.3 产品目标

帮助用户解决以下问题：

1. 岗位 JD 看不懂，不知道该准备什么
2. 简历和目标岗位脱节，不知道该怎么改
3. 面试资料太散，知识点和题库难以消化
4. 不知道改完简历后最可能被问什么

### 2.4 产品定位

这不是：

- 一个泛用聊天机器人
- 一个只有“简历优化”的单点工具
- 一个万能求职平台

这是：

- 一个聚焦 AI/产品岗求职场景的 `垂直 AI 产品`
- 一个把 `简历`、`JD`、`预置面经知识库` 和 `模拟面试` 串起来的 `中文求职准备工作台`

## 3. 目标用户

### 3.1 核心用户

第一阶段只服务一类用户：

**正在准备 AI 产品经理 / 产品经理 / AI 应用类岗位面试的求职者**

### 3.2 用户特征

- 正在投递 AI/产品相关岗位
- 具备基本信息检索能力
- 已经收集了一些 JD、面经和知识资料
- 最大痛点不是“没资料”，而是“资料太多但不会用，也不会围绕岗位改简历”

### 3.3 首批用户

- 你自己
- 和你类似的校招/实习求职者

## 4. 用户痛点与需求

### 4.1 典型痛点

1. 面试知识点分散，难以系统准备
2. 每个岗位都在重复搜资料、重复整理
3. 简历和 JD 的匹配关系不清楚
4. 不知道哪些项目经历该强化、该怎么写
5. 真实面试时不知道该如何结构化回答
6. 不知道改完简历后最可能被追问什么

### 4.2 核心需求

1. 快速理解一个岗位最可能考什么
2. 能让简历围绕目标岗位完成针对性改写
3. 能把平台预置知识库融合进简历包装和答题准备
4. 能输出高概率问题和答题辅助
5. 后续再进入更完整的模拟面试和记录沉淀

## 5. 核心用户路径

### 5.1 主路径

1. 用户登录账号
2. 用户上传并保存 PDF 简历
3. 系统解析并保存结构化简历
4. 用户粘贴岗位 JD 并记录来源 URL
5. 系统按优先级生成改写建议：
   - 第一优先级：紧扣岗位 JD
   - 第二优先级：参考平台知识库
6. 用户查看建议并进入模拟面试

### 5.2 价值闭环

用户不是只“问了一次 AI”，而是完成：

`登录 -> 简历归档 -> 岗位理解 -> 简历改写 -> 模拟面试`

这就是产品的核心闭环。

## 6. MVP 范围

### 6.1 当前 MVP 只做 4 个用户核心能力

1. `登录后保存并复用简历`
2. `保存目标 JD 并记录来源 URL`
3. `基于 JD + 平台知识库的简历改写建议`
4. `基于高频面经的模拟面试`

支撑能力：

- `平台预置知识库 / RAG`
- `Workflow`
- `Memory`
- `Prompt`
- `轻 Agent（后续用于多轮模拟）`
- 所有用户可见页面统一使用中文

### 6.2 MVP 不做

- 自动投递
- 岗位爬虫平台
- 视频/语音面试分析
- 社区和社交功能
- 多岗位泛化推荐系统
- 高复杂度 Agent 全自动流程
- 让用户自己折腾 RAG 配置

### 6.3 MVP 验证目标

这版 MVP 只验证 3 件事：

1. 用户是否愿意上传并持续复用自己的简历
2. 用户是否愿意围绕目标 JD 查看改写建议并进入模拟面试
3. 平台预置知识库 + Workflow + Prompt 是否能形成可用的中文求职准备闭环

## 7. 功能需求

## 7.1 模块一：登录后保存并复用简历

### 功能目标

把用户原始简历沉淀到账号中，默认作为后续所有岗位准备的基础。

### 输入

- PDF 简历文件

### 输出

- 当前主简历
- 结构化简历摘要
- 项目经历要点
- 可强化表达点

### 用户价值

让系统后续不是围绕一段原始长文工作，而是围绕结构化简历上下文进行改写和面试辅助。

### 功能细项

1. 支持登录后上传并保存 PDF 简历
2. 自动抽取项目经历与角色信息
3. 默认复用当前主简历
4. 支持用户手动替换简历版本

## 7.2 模块二：保存目标 JD 并记录来源 URL

### 功能目标

将目标岗位 JD 变成可复用的岗位卡片和结构化准备清单。

### 输入

- 岗位名称
- 公司名称
- JD 文本
- 来源 URL

### 输出

- 已保存岗位卡片
- 岗位能力标签
- 重点考察维度
- 高频题型推荐
- 推荐准备顺序

### 用户价值

让用户从“看了一段 JD 但没感觉”变成“知道该准备哪些内容”。

### 功能细项

1. 保存岗位来源 URL
2. 提取岗位关键词
2. 判断岗位偏向：
   - AI 基础
   - 产品设计
   - 场景题
   - 指标评测
   - 业务理解
3. 输出 5-10 个高频考察点
4. 生成推荐准备清单
5. 支持用户后续重复选择该 JD

## 7.3 模块三：基于 JD + 平台知识库的简历改写建议

### 功能目标

让用户围绕当前岗位看到可执行的改写建议，而不是被强制重写简历。

### 输入

- 用户当前主简历
- 目标岗位 JD
- 平台预置面经 / 常见问题 / 知识点

### 输出

- 改写建议
- 推荐表达方式
- 每段经历的岗位匹配理由
- 可能引出的高频问题

### 用户价值

让用户不是只“润色简历”，而是把简历改成更贴岗位、更利于面试展开的版本。

### 功能细项

1. 第一优先级：围绕目标 JD 调整表达
2. 第二优先级：结合平台知识库调整项目表达
3. 输出修改理由，而不是只给结果
4. 提示哪些修改会引出哪些高概率面试点

## 7.4 模块四：基于高频面经的模拟面试

### 功能目标

帮助用户围绕当前岗位和当前简历进行定向模拟面试。

### 输入

- 目标岗位
- 当前简历
- 改写建议重点
- 平台预置知识库

### 输出

- 当前面试问题
- 可能追问点
- 答题思路
- 多轮问答上下文

### 题型范围

1. 动机题
2. AI 基础题
3. 项目深挖题
4. 场景设计题
5. 指标/评测题

### 功能细项

1. 生成高频问题
2. 基于用户回答追问
3. 输出每道题的答题要点
4. 结合知识库给出辅助解释

## 7.5 支撑能力：平台预置知识库

### 功能目标

让用户无需自己配置 RAG，也能直接使用面经和知识体系。

### 知识源

- 平台维护的面经文档
- 常见面试问题
- AI 产品知识笔记
- 用户自己的简历 / JD / 项目资料

### 用户价值

让产品开箱即用，并随着知识库积累不断变强。

## 8. 信息架构

建议 MVP 信息架构如下：

1. `登录页`
2. `单工作台`
3. `历史记录（后续）`
4. `设置（后续）`

### 8.1 登录页

展示：

- 邮箱 / Google / GitHub 登录入口
- 产品说明

### 8.2 单工作台

单工作台按 4 步组织：

1. 我的简历
2. 目标 JD
3. 改写建议
4. 模拟面试

要求：

- 全中文文案
- 尽量减少页面跳转
- RAG 对用户隐藏，不暴露为独立主页面

工作台内部视图：

#### Step 1 我的简历

展示：

- 当前主简历
- 简历摘要
- 项目经历要点
- 可强化表达点

#### Step 2 目标 JD

展示：

- 原始 JD
- 来源 URL
- 拆解后的能力维度
- 推荐准备清单

#### Step 3 改写建议

展示：

- 原始简历和建议表达对照
- 每条修改建议
- 修改理由
- 对应岗位匹配点

#### Step 4 模拟面试

展示：

- 当前问题
- 追问点
- 答题思路
- 相关知识来源

## 9. AI 方案设计

## 9.1 整体原则

不是把产品做成一个“会聊天的壳”，而是明确区分：

- 哪些环节适合 `Workflow`
- 哪些环节适合 `RAG`
- 哪些环节适合 `状态记忆`
- 哪些环节可以做成 `轻 Agent`

## 9.2 RAG 适用环节

适合用 RAG 的模块：

- 平台预置面经检索
- 岗位解析辅助
- 简历改写建议
- 高频问题与答题辅助

RAG 的作用：

- 用已有资料回答问题
- 降低幻觉
- 提高针对性

## 9.3 Workflow 适用环节

适合用 Workflow 的模块：

- JD 拆解
- 简历结构化
- 简历按 JD 改写
- 问题生成与答题辅助

原因：

- 任务高频
- 输出格式稳定
- 便于控制和评测

## 9.4 Memory / 状态适用环节

需要记录两类信息：

### 用户长期信息

- 目标岗位
- 目标城市
- 简历版本
- 项目列表
- 已完成的岗位改写记录

### 当前任务状态

- 当前正在准备哪个岗位
- 当前正在改哪份简历
- 最近一次改写结果
- 推荐下一步动作

这部分能让产品具备“持续准备”的体验，而不是单轮对话。

## 9.5 轻 Agent 适用环节

适合做成轻 Agent 的模块：

- 后续多轮模拟面试
- 根据当前岗位和简历动态追问
- 根据用户表现调整题目顺序

为什么是“轻 Agent”：

- 目标明确
- 路径半开放
- 仍然可控

不做全自动通用 Agent，是为了降低复杂度和不稳定性。

## 10. 评分与评测设计

## 10.1 回答评分 rubric

建议评分维度：

1. `切题程度`
2. `结构清晰度`
3. `AI 理解深度`
4. `产品思维`
5. `业务感`

每项 1-5 分，总分 25。

## 10.2 知识问答评测

可关注：

- 回答覆盖率
- 结构完整度
- 与知识库相关性
- 无法回答时的兜底表现

## 10.3 产品价值评测

核心不是“模型答得像不像”，而是：

- 用户是否完成一轮完整训练
- 用户是否愿意继续使用
- 用户是否能从反馈中优化答案

## 11. 指标设计

## 11.1 北极星指标

推荐：

**每周完成有效岗位准备任务的用户数**

为什么不用：

- 对话次数
- 问题数量

因为这些都不能直接代表用户真的准备得更好了。

## 11.2 核心过程指标

1. 岗位解析完成率
2. 简历改写完成率
3. 面试辅助使用率
4. 完整准备闭环完成率
5. 次日/7日复用率

## 11.3 护栏指标

1. 明显错误建议率
2. 幻觉率
3. 用户中途放弃率
4. 平均响应时长
5. 用户负向反馈率

## 12. 非功能要求

### 12.1 响应速度

MVP 阶段应尽量控制核心交互在可接受范围内，避免因响应过慢让用户中断训练。

### 12.2 可解释性

对于评分、改写建议和知识问答，应尽量保证：

- 结构化输出
- 说明依据
- 出现不确定性时明确说明

### 12.3 可扩展性

虽然当前只做文字版 MVP，但数据结构上应预留未来扩展：

- 语音面试
- 视频面试
- 多岗位类型

## 13. 风险与对策

### 风险 1：简历改写过度润色或不贴用户真实经历

对策：

- 保留原始简历和改写结果对照
- 输出每条修改理由
- 明确只做岗位匹配增强，不捏造经历

### 风险 2：知识库回答不准确

对策：

- 优化知识切分
- 控制上下文长度
- 给出无法确认时的兜底回答

### 风险 3：产品范围失控

对策：

- 只聚焦 AI/产品岗
- 只做文字版 MVP
- 只保留 4 个核心模块

## 14. 用户故事

### 用户故事 1

作为一个准备 AI 产品岗面试的用户，  
我希望输入 JD 后能快速知道重点考察什么，  
这样我就不用自己从零拆岗位要求。

### 用户故事 2

作为一个已经有目标 JD 的用户，
我希望系统能结合岗位要求和平台知识库帮我改简历，
这样我就不用自己猜哪些项目点该怎么写。

### 用户故事 3

作为一个准备面试的用户，  
我希望看到高概率问题、追问点和答题思路，
这样我能围绕当前岗位更有针对性地准备。

### 用户故事 4

作为一个反复投递和修改简历的用户，
我希望系统能记住我当前在准备哪个岗位、改过哪些内容，
这样我每次打开都能直接进入下一步。

## 15. 优先级

### P0

- 简历输入与结构化
- JD 解析
- 简历按 JD 改写
- 面试辅助
- 平台预置知识库

### P1

- 更完整的模拟面试
- 个性化追问
- 记录与薄弱项追踪

### P2

- 语音输入
- 视频面试
- 社区分享

## 16. 上线标准

MVP 上线前至少满足：

1. 可以稳定完成 JD 解析
2. 简历可围绕岗位完成针对性改写
3. 面试辅助能输出高概率问题与答题建议
4. 知识库引用范围清晰
5. 用户简历与岗位上下文能被保存和复用

## 17. 成功定义

这个项目成功，不是指它变成一个完整商业产品，而是：

1. 能形成可演示的完整闭环
2. 能写进简历
3. 能在面试中讲清楚：
   - 为什么做
   - 用户是谁
   - 为什么用 AI
   - 为什么用 RAG / Workflow / Memory
   - 如何评测
   - 如何定义指标

## 18. 下一步

PRD v1 之后，下一步建议按这个顺序推进：

1. 完成文档对齐
2. 建简历工作区与结构化流程
3. 实现简历按 JD 改写 workflow
4. 接入面试辅助输出
5. 再进入更完整的模拟面试
