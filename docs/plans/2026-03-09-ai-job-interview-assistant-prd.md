# AI 求职面试助手 PRD v1

## 1. 文档目的

本文档用于定义 `AI 求职面试助手` 的第一版产品方案，明确：

- 产品目标
- 核心用户
- 核心场景
- MVP 功能范围
- 信息架构
- AI 方案拆解
- 指标与评测
- 非目标范围

这版 PRD 的目标不是追求“大而全”，而是指导一个可在短时间内完成、能写进简历、能讲进面试的 MVP。

## 2. 产品概述

### 2.1 产品名称

暂定名：

- `AI 求职面试助手`
- 英文工作名：`OfferPilot`

### 2.2 一句话定义

**一个面向 AI/产品岗求职者的 AI 面试准备 Copilot，帮助用户把 `JD 解析 -> 知识学习 -> 模拟面试 -> 薄弱项追踪` 串成闭环。**

### 2.3 产品目标

帮助用户解决以下问题：

1. 岗位 JD 看不懂，不知道该准备什么
2. 面试资料太散，知识点和题库难以消化
3. 会看不会答，缺少高质量模拟练习
4. 练过的问题没有沉淀，薄弱项无法持续追踪

### 2.4 产品定位

这不是：

- 一个泛用聊天机器人
- 一个只有“简历优化”的单点工具
- 一个万能求职平台

这是：

- 一个聚焦 AI/产品岗求职场景的 `垂直 AI 产品`
- 一个把知识库、题库、模拟和复盘打通的 `求职训练系统`

## 3. 目标用户

### 3.1 核心用户

第一阶段只服务一类用户：

**正在准备 AI 产品经理 / 产品经理 / AI 应用类岗位面试的求职者**

### 3.2 用户特征

- 正在投递 AI/产品相关岗位
- 具备基本信息检索能力
- 已经收集了一些 JD、面经和知识资料
- 最大痛点不是“没资料”，而是“资料太多但不会用”

### 3.3 首批用户

- 你自己
- 和你类似的校招/实习求职者

## 4. 用户痛点与需求

### 4.1 典型痛点

1. 面试知识点分散，难以系统准备
2. 每个岗位都在重复搜资料、重复整理
3. 简历和 JD 的匹配关系不清楚
4. 真实面试时不知道该如何结构化回答
5. 模拟面试缺少反馈，练习难形成闭环
6. 不知道自己到底薄弱在哪

### 4.2 核心需求

1. 快速理解一个岗位最可能考什么
2. 能基于目标岗位生成个性化题库
3. 能对知识点进行结构化问答
4. 能模拟面试并获得反馈
5. 能记录历史练习和薄弱项

## 5. 核心用户路径

### 5.1 主路径

1. 用户粘贴岗位 JD
2. 系统拆解能力要求与高频题型
3. 用户导入简历与个人项目
4. 系统生成个性化题库与推荐准备重点
5. 用户进入模拟面试
6. 用户提交回答
7. 系统输出结构化反馈与改写建议
8. 系统记录薄弱项并推荐下一轮练习

### 5.2 价值闭环

用户不是只“问了一次 AI”，而是完成：

`岗位理解 -> 练习 -> 反馈 -> 复盘 -> 再练习`

这就是产品的核心闭环。

## 6. MVP 范围

### 6.1 MVP 只做 4 个模块

1. `JD 解析`
2. `知识库问答`
3. `模拟面试`
4. `练习记录与薄弱项追踪`

### 6.2 MVP 不做

- 自动投递
- 岗位爬虫平台
- 视频/语音面试分析
- 社区和社交功能
- 多岗位泛化推荐系统
- 高复杂度 Agent 全自动流程

### 6.3 MVP 验证目标

这版 MVP 只验证 3 件事：

1. 用户是否能更快理解一个岗位该准备什么
2. 用户是否愿意持续用它做模拟和复盘
3. RAG + Workflow + 状态追踪是否能形成有效训练闭环

## 7. 功能需求

## 7.1 模块一：JD 解析

### 功能目标

将原始 JD 转换成结构化准备清单。

### 输入

- 岗位名称
- 公司名称
- JD 文本

### 输出

- 岗位能力标签
- 重点考察维度
- 高频题型推荐
- 推荐准备顺序

### 用户价值

让用户从“看了一段 JD 但没感觉”变成“知道该准备哪些内容”。

### 功能细项

1. 提取岗位关键词
2. 判断岗位偏向：
   - AI 基础
   - 产品设计
   - 场景题
   - 指标评测
   - 业务理解
3. 输出 5-10 个高频考察点
4. 生成推荐准备清单

## 7.2 模块二：知识库问答

### 功能目标

让用户围绕已有资料进行结构化提问。

### 知识源

- 面经文档
- 岗位 JD
- AI 产品知识笔记
- 用户自己的项目文档

### 典型问题

- “RAG 的作用是什么？”
- “这个岗位为什么会考 Agent？”
- “怎么解释北极星指标？”
- “我的这个项目能怎么包装进简历？”

### 功能细项

1. 支持自然语言提问
2. 从知识库检索相关内容
3. 输出结构化回答
4. 在回答中提示依据来源或知识范围

### 用户价值

让用户不是“到处翻资料”，而是可以围绕目标岗位快速学习。

## 7.3 模块三：模拟面试

### 功能目标

帮助用户把“知道知识点”变成“能说出来”。

### 输入

- 目标岗位
- 题型偏好
- 用户回答文本

### 输出

- 当前题目
- 结构化评分
- 问题点分析
- 改写建议

### 题型范围

1. 动机题
2. AI 基础题
3. 项目深挖题
4. 场景设计题
5. 指标/评测题

### 功能细项

1. 生成问题
2. 接收用户回答
3. 按 rubric 评分
4. 输出反馈
5. 推荐追问或下一题

## 7.4 模块四：练习记录与薄弱项追踪

### 功能目标

帮助用户形成持续训练闭环。

### 记录内容

- 已练题目
- 题型分布
- 历次评分
- 高频错误点
- 推荐补课方向

### 功能细项

1. 保存练习记录
2. 按题型汇总表现
3. 标注薄弱项
4. 给出下一轮建议

### 用户价值

让用户能清楚看到自己“还差什么”，而不是每次都从头练。

## 8. 信息架构

建议 MVP 信息架构如下：

1. `首页`
2. `岗位解析页`
3. `知识库问答页`
4. `模拟面试页`
5. `复盘页`

### 8.1 首页

展示：

- 当前目标岗位
- 推荐下一步动作
- 最近练习记录
- 当前薄弱项

### 8.2 岗位解析页

展示：

- 原始 JD
- 拆解后的能力维度
- 推荐问题清单
- 推荐知识点

### 8.3 知识库问答页

展示：

- 输入框
- 回答区
- 相关知识来源

### 8.4 模拟面试页

展示：

- 当前问题
- 回答输入区
- 提交按钮
- 评分反馈区

### 8.5 复盘页

展示：

- 题型表现
- 薄弱项标签
- 建议下一步

## 9. AI 方案设计

## 9.1 整体原则

不是把产品做成一个“会聊天的壳”，而是明确区分：

- 哪些环节适合 `Workflow`
- 哪些环节适合 `RAG`
- 哪些环节适合 `状态记忆`
- 哪些环节可以做成 `轻 Agent`

## 9.2 RAG 适用环节

适合用 RAG 的模块：

- 知识库问答
- 岗位解析辅助
- 项目包装建议
- 高频题解释

RAG 的作用：

- 用已有资料回答问题
- 降低幻觉
- 提高针对性

## 9.3 Workflow 适用环节

适合用 Workflow 的模块：

- JD 拆解
- 题型归类
- 回答评分
- 复盘总结

原因：

- 任务高频
- 输出格式稳定
- 便于控制和评测

## 9.4 Memory / 状态适用环节

需要记录两类信息：

### 用户长期信息

- 目标岗位
- 目标城市
- 简历版本
- 项目列表
- 偏弱题型

### 当前任务状态

- 当前练到哪一题
- 上一次评分结果
- 已经做过哪些练习
- 推荐下一步动作

这部分能让产品具备“持续训练”的体验，而不是单轮对话。

## 9.5 轻 Agent 适用环节

适合做成轻 Agent 的模块：

- 多轮模拟面试
- 根据前文表现动态追问
- 根据薄弱项调整题目顺序

为什么是“轻 Agent”：

- 目标明确
- 路径半开放
- 仍然可控

不做全自动通用 Agent，是为了降低复杂度和不稳定性。

## 10. 评分与评测设计

## 10.1 回答评分 rubric

建议评分维度：

1. `切题程度`
2. `结构清晰度`
3. `AI 理解深度`
4. `产品思维`
5. `业务感`

每项 1-5 分，总分 25。

## 10.2 知识问答评测

可关注：

- 回答覆盖率
- 结构完整度
- 与知识库相关性
- 无法回答时的兜底表现

## 10.3 产品价值评测

核心不是“模型答得像不像”，而是：

- 用户是否完成一轮完整训练
- 用户是否愿意继续使用
- 用户是否能从反馈中优化答案

## 11. 指标设计

## 11.1 北极星指标

推荐：

**每周完成有效面试训练任务的用户数**

为什么不用：

- 对话次数
- 问题数量

因为这些都不能直接代表用户真的准备得更好了。

## 11.2 核心过程指标

1. 岗位解析完成率
2. 知识问答使用率
3. 模拟面试提交率
4. 完整训练闭环完成率
5. 次日/7日复练率

## 11.3 护栏指标

1. 明显错误回答率
2. 幻觉率
3. 用户中途放弃率
4. 平均响应时长
5. 用户负向反馈率

## 12. 非功能要求

### 12.1 响应速度

MVP 阶段应尽量控制核心交互在可接受范围内，避免因响应过慢让用户中断训练。

### 12.2 可解释性

对于评分、改写建议和知识问答，应尽量保证：

- 结构化输出
- 说明依据
- 出现不确定性时明确说明

### 12.3 可扩展性

虽然当前只做文字版 MVP，但数据结构上应预留未来扩展：

- 语音面试
- 视频面试
- 多岗位类型

## 13. 风险与对策

### 风险 1：AI 评分不稳定

对策：

- 结构化 rubric
- 尽量减少开放式评价
- 明确评分仅作训练参考

### 风险 2：知识库回答不准确

对策：

- 优化知识切分
- 控制上下文长度
- 给出无法确认时的兜底回答

### 风险 3：产品范围失控

对策：

- 只聚焦 AI/产品岗
- 只做文字版 MVP
- 只保留 4 个核心模块

## 14. 用户故事

### 用户故事 1

作为一个准备 AI 产品岗面试的用户，  
我希望输入 JD 后能快速知道重点考察什么，  
这样我就不用自己从零拆岗位要求。

### 用户故事 2

作为一个正在整理知识点的用户，  
我希望能直接对面经和知识库提问，  
这样我就不用来回翻文档。

### 用户故事 3

作为一个准备面试的用户，  
我希望能进行模拟问答并获得结构化反馈，  
这样我能知道自己到底哪里答得不好。

### 用户故事 4

作为一个反复练习的用户，  
我希望系统能记住我练过什么和薄弱点，  
这样我每次打开都能直接进入下一步。

## 15. 优先级

### P0

- JD 解析
- 知识库问答
- 模拟面试
- 评分反馈
- 练习记录

### P1

- 薄弱项自动归类
- 题型推荐
- 个性化追问

### P2

- 语音输入
- 视频面试
- 社区分享

## 16. 上线标准

MVP 上线前至少满足：

1. 可以稳定完成 JD 解析
2. 知识库问答能回答核心问题
3. 模拟面试流程可跑通
4. 回答评分输出结构清晰
5. 用户练习记录能被保存和展示

## 17. 成功定义

这个项目成功，不是指它变成一个完整商业产品，而是：

1. 能形成可演示的完整闭环
2. 能写进简历
3. 能在面试中讲清楚：
   - 为什么做
   - 用户是谁
   - 为什么用 AI
   - 为什么用 RAG / Workflow / Memory
   - 如何评测
   - 如何定义指标

## 18. 下一步

PRD v1 之后，下一步建议按这个顺序推进：

1. 产出页面级功能清单
2. 定义知识库结构
3. 定义状态与数据结构
4. 确定技术栈
5. 开始实现 MVP
